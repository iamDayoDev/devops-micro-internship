# Ansible Azure Ad-Hoc Lab

This project provisions Azure infrastructure with Terraform and uses Ansible ad-hoc commands to manage the provisioned Linux hosts without writing a playbook.

The repository currently defines a **2-VM lab**, not a 4-VM lab:

- A **web VM** in a public subnet with a public IP
- An **app VM** in a private subnet with no public IP

SSH access is configured with an **ED25519 key pair**, password authentication is disabled, and Ansible reaches the private app host through the web VM by using a **jump host** configuration.

## Architecture Overview

Terraform creates the following Azure resources:

- A Resource Group: `rg-ansible-lab`
- A Virtual Network: `10.0.0.0/16`
- A public subnet: `10.0.1.0/24`
- A private subnet: `10.0.2.0/24`
- A Network Security Group allowing:
  - SSH on port `22`
  - HTTP on port `80`
- One public IP for the web VM
- One network interface for the web VM
- One network interface for the app VM
- One Ubuntu 22.04 web VM
- One Ubuntu 22.04 app VM

Authentication is key-based only:

- `disable_password_authentication = true`
- `admin_ssh_key` injects the public key into each VM


### What Each File Does

- `providers.tf`
  Configures Terraform itself and the AzureRM provider.

- `variables.tf`
  Stores configurable inputs such as Azure region, admin username, VM size, tags, VM role names, and the SSH public key path.

- `main.tf`
  Defines the Azure infrastructure: resource group, VNet, subnets, NSG, NICs, public IP, and the two Linux virtual machines.

- `outputs.tf`
  Exposes useful values after deployment, including:
  - `web_vm_public_ip`
  - `web_vm_private_ip`
  - `app_vm_private_ip`
  - `web_ssh_command`

- `ansible/inventory.ini`
  Defines the Ansible inventory groups and connection settings for both hosts.

- `README.md`
  Documents how the project works, how SSH is configured, and how to run Ansible ad-hoc commands.

## SSH Key Setup

This project uses SSH key authentication instead of passwords.

### 1. Generate an SSH Key If Needed

If you do not already have an ED25519 key pair, create one:
I created rsa because ed25519 is not supporteed on Microsoft Azure VMs

```bash
ssh-keygen -t rsa -f ~/.ssh/id_ed25519 -C "azure-ansible-lab"
```

This creates:

- Private key: `~/.ssh/id_ed25519`
- Public key: `~/.ssh/id_ed25519.pub`

### 2. How Terraform Uses the Key

The public key path is defined in `variables.tf`:

```hcl
variable "ssh_public_key" {
  description = "Path to the SSH public key used for admin access to the Linux VMs."
  type        = string
  default     = "~/.ssh/id_ed25519.pub"
}
```

Terraform reads that file in the VM resource:

```hcl
admin_ssh_key {
  username   = var.admin_username
  public_key = file(var.ssh_public_key)
}
```

Because both VMs also set:

```hcl
disable_password_authentication = true
```

the VMs accept SSH access through your key only.

### 3. Important Note About Key Paths

If `~/.ssh/id_ed25519.pub` does not resolve correctly on your machine, set an explicit path instead by using a `terraform.tfvars` file or a CLI variable:

```hcl
ssh_public_key = "C:/Users/USER/.ssh/id_ed25519.pub"
```

## Validating SSH Access

### Access the Web VM

The web VM has a public IP, so you can connect directly:

```bash
ssh azureuser@<web_vm_public_ip> "hostname"
```

Expected result:

- The command returns the hostname of the web VM
- No password prompt appears

### Access the App VM

The app VM is on the private subnet and does not have a public IP. To reach it, connect through the web VM using `ProxyJump`:

```bash
ssh -J azureuser@<web_vm_public_ip> azureuser@<app_vm_private_ip> "hostname"
```

This proves:

- The SSH key was injected successfully
- The web VM can act as a jump box
- The private app VM is reachable securely without exposing it directly to the internet

## Ansible Inventory

The inventory file lives at:

```text
ansible/inventory.ini
```

### Inventory Design

This project groups hosts by role:

- `[web]` contains the publicly reachable web VM
- `[app]` contains the private app VM
- `[all:vars]` defines the shared SSH username and private key
- `[app:vars]` adds the jump-host setting required to reach the private VM

### Example Inventory Layout

```ini
[web]
<web_vm_public_ip>

[app]
<app_vm_private_ip>

[all:vars]
ansible_user=azureuser
ansible_ssh_private_key_file=~/.ssh/id_ed25519

[app:vars]
ansible_ssh_common_args=-o ProxyJump=azureuser@<web_vm_public_ip>
```

### Why `ProxyJump` Is Needed

The app server sits in the private subnet and has no public IP. Ansible therefore cannot connect to it directly from your control machine.

This line solves that:

```ini
ansible_ssh_common_args=-o ProxyJump=azureuser@<web_vm_public_ip>
```

It tells Ansible to:

1. SSH into the public web VM first
2. Use that host as the jump box
3. Continue to the private app VM

## Updating the Inventory After Terraform Apply

After provisioning, update `ansible/inventory.ini` with the latest values from Terraform outputs:

- Put the web VM public IP under `[web]`
- Put the app VM private IP under `[app]`
- Update the `ProxyJump` target to the web VM public IP

You can view the values with:

```bash
terraform output web_vm_public_ip
terraform output app_vm_private_ip
terraform output web_vm_private_ip
```

## Running Ansible Ad-Hoc Commands

All commands below are run from the project root or from inside `ansible-azure-adhoc`.

### 1. Test Connectivity to All Hosts

```bash
ansible all -i ansible/inventory.ini -m ping
```

This confirms:

- SSH connectivity works
- The inventory is correct
- The SSH key is being used properly
- The jump-host configuration for the app server works

### 2. Check the Remote User

```bash
ansible all -i ansible/inventory.ini -m command -a "whoami"
```

Expected result:

- Both hosts should return `azureuser`

### 3. Check Uptime on All Hosts

```bash
ansible all -i ansible/inventory.ini -m command -a "uptime"
```

This is a quick fleet-wide health check.

### 4. Display Hostnames

```bash
ansible all -i ansible/inventory.ini -m command -a "hostname"
```

Useful when you want to verify which machine answered.

### 5. Install Nginx on the Web Host

```bash
ansible web -i ansible/inventory.ini -m apt -a "name=nginx state=present update_cache=yes" --become
```

This installs Nginx only on the web VM.

### 6. Start and Enable Nginx on the Web Host

```bash
ansible web -i ansible/inventory.ini -m service -a "name=nginx state=started enabled=yes" --become
```

This ensures:

- Nginx is running now
- Nginx starts automatically on reboot

### 7. Install `htop` on Both Hosts

```bash
ansible all -i ansible/inventory.ini -m apt -a "name=htop state=present update_cache=yes" --become
```

This demonstrates a simple fleet-wide software installation.

### 8. Check Disk Usage on the App Host

```bash
ansible app -i ansible/inventory.ini -m command -a "df -h"
```

This targets the private application host only.

### 9. Check Memory Usage on All Hosts

```bash
ansible all -i ansible/inventory.ini -m command -a "free -m"
```

### 10. Run a Command on the Web Group Only

```bash
ansible web -i ansible/inventory.ini -m command -a "systemctl status nginx --no-pager"
```

## Why Ad-Hoc Commands Matter in This Project

This repo is focused on quick operational tasks without writing a full playbook. Ad-hoc commands are useful when you want to:

- Verify connectivity
- Check host state quickly
- Install one package across one or more hosts
- Start or enable a service
- Run inspection commands across groups

They are ideal for fast checks in a small lab or sandbox environment.

## End-to-End Workflow

The full workflow for this project is:

1. Authenticate to Azure with `az login`
2. Ensure your SSH key pair exists
3. Run `terraform init`
4. Run `terraform apply`
5. Read the Terraform outputs
6. Update `ansible/inventory.ini`
7. Test SSH access
8. Run Ansible ad-hoc commands against `web`, `app`, or `all`

## Cleanup

To remove all Azure resources created by this project:

```bash
terraform destroy
```
